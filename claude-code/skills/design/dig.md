# Dig スキル

## Overview

要件の曖昧な点を掘り下げ、設計を固め、承認を経てスペック文書を出力するスキル。実装に入る前の「考える段階」を構造化する。

**Announce at start:** "Dig スキルで要件を掘り下げ、設計を固めます。"

<HARD-GATE>
このスキルが完了するまで、実装スキル（/tdd, /task-decompose 等）の呼び出し、実装コードの記述、実装ファイル/ディレクトリのスキャフォールドを行ってはならない。設計が承認され、スペック文書が書かれた後にのみ実装に進む。
「シンプルだから設計不要」は禁止。設計は短くてよい（数行でも可）が、必ず提示して承認を得ること。

**本スキル内で許容される出力 (例外):**
- Phase 2.5 の Visual Companion HTML (`/tmp/dig-visual/*.html`)
- Phase 6 のスペック文書 (`docs/specs/YYYY-MM-DD-<topic>-design.md`)
- 既存ソースに対する Read / Glob / Grep / Bash 等の調査系ツール
</HARD-GATE>

## 使用方法

```
/dig                          # 現在のプランや直近の会話から曖昧な点を掘り下げ
/dig <機能や計画の説明>        # 指定した内容について掘り下げ
/dig release/feature-name     # リリース仕様書を読んで曖昧な点を掘り下げ
```

## ワークフロー

```
Phase 1: コンテキスト収集
    ↓
Phase 2: 曖昧点の特定
    ↓
Phase 2.5: Visual Companion 提案（ビジュアル質問がありそうな場合）
    ↓
Phase 3: 構造化された質問（1問ずつ、必要に応じてビジュアル）
    ↓
Phase 4: アプローチ提案（2-3案 + トレードオフ）
    ↓
Phase 5: 設計提示 & 承認ゲート
    ↓
Phase 6: スペック文書の出力 & セルフレビュー
    ↓
Phase 7: ユーザーによるスペックレビュー
    ↓
Phase 8: 実装ハンドオフ
```

## Phase 1: コンテキスト収集

以下のファイルを読み込んでプロジェクトの文脈を把握する：

1. **CLAUDE.md** --- プロジェクトのアーキテクチャ・規約
2. **リリース仕様書** --- 引数で指定された場合、`.claude/release-specs/{name}.md` を読む
3. **プランファイル** --- `.claude/plans/` の該当ファイル
4. **関連する既存コード** --- Glob/Grep で対象領域を調査
5. **最近のコミット** --- `git log --oneline -10` で直近の変更を把握

引数なしの場合は、直近の会話コンテキストから対象を推定する。

## Phase 2: 曖昧点の特定

以下のカテゴリで曖昧な点を洗い出す。プロジェクトの特性に応じて重点カテゴリを選ぶ：

| カテゴリ | 着眼点 |
|---------|--------|
| **スコープ** | 含めるもの/含めないもの、将来対応に回すもの |
| **アーキテクチャ** | コンポーネント構成、責務分離、レイヤー設計 |
| **データ** | テーブル設計、スキーマ、データアクセスパターン |
| **API** | エンドポイント設計、リクエスト/レスポンス型、エラーハンドリング |
| **UI/UX** | 画面遷移、レイアウト、レスポンシブ、アクセシビリティ |
| **テスト** | テスト戦略、カバレッジ範囲、E2E の範囲 |
| **セキュリティ / 認証** | 認証フロー、権限、入力バリデーション |

### スコープ判定

曖昧点を洗い出す前に、まずスコープを判定する：
- 要求が**複数の独立したサブシステム**を含む場合、先にサブプロジェクトに分解する
- 各サブプロジェクトが独自の dig → スペック → 実装サイクルを持つ
- 1つのスペックで扱うには大きすぎないか常に意識する

### 分析ルール

- 明示されていない仮定を洗い出す
- 既存コードとの整合性を確認する
- 「どちらでも良い」ではなく、具体的なトレードオフを示す
- 曖昧点が見つからないカテゴリはスキップする

## Phase 2.5: Visual Companion（Playwright MCP）

今後の質問に**ビジュアルで見せた方が伝わる内容**（UI モックアップ、レイアウト比較、アーキテクチャ図など）が含まれそうな場合、Visual Companion の使用を提案する。

### 提案メッセージ（独立したメッセージとして送る）

> "この設計では UI やレイアウトに関する質問がありそうです。ブラウザでモックアップや図を見ながら進めることもできます。Playwright でブラウザに表示しますか？"

**この提案は単独のメッセージで行う。** 他の質問や説明と混ぜない。ユーザーが断ったらテキストのみで進行する。

### 質問ごとの判断基準

Visual Companion を有効にした後も、**質問ごとに**ブラウザを使うか判断する:

**ブラウザを使う場合:**
- UI モックアップ・ワイヤーフレーム
- レイアウトの A/B 比較（side-by-side）
- アーキテクチャ図・データフロー図
- デザインの選択肢を視覚的に提示する場合

**ターミナルで十分な場合:**
- 要件やスコープの質問
- コンセプトの選択（言葉で説明できるもの）
- トレードオフのリスト
- 技術的判断（API 設計、データモデリング等）

UI トピックについての質問でも、自動的にビジュアルにしない。「ウィザード形式にしますか？」は概念的質問 → ターミナル。「このウィザードのレイアウトどちらが良い？」はビジュアル質問 → ブラウザ。

### 使い方（Playwright MCP）

1. **HTML ファイルを書き出す**: `/tmp/dig-visual/` 以下にセマンティックなファイル名で書き出す

```html
<!-- /tmp/dig-visual/layout-comparison.html -->
<!DOCTYPE html>
<html><head>
<meta charset="utf-8">
<style>
  body { font-family: system-ui; background: #1a1a2e; color: #e0e0e0; padding: 2rem; }
  .question { font-size: 1.2rem; margin-bottom: 1.5rem; }
  .options { display: flex; gap: 1.5rem; flex-wrap: wrap; }
  .option { flex: 1; min-width: 300px; border: 2px solid #333; border-radius: 12px;
            padding: 1.5rem; cursor: pointer; transition: border-color 0.2s; }
  .option:hover { border-color: #6c63ff; }
  .option h3 { margin-top: 0; color: #6c63ff; }
  .mockup { background: #16213e; border-radius: 8px; padding: 1rem; min-height: 200px; }
  .split { display: flex; gap: 1rem; }
  .split > * { flex: 1; }
  .label { font-size: 0.75rem; text-transform: uppercase; color: #888; letter-spacing: 0.05em; }
</style>
</head><body>
  <!-- 質問とオプションをここに -->
</body></html>
```

2. **Playwright で開く**: `browser_navigate` で `file:///tmp/dig-visual/layout-comparison.html` を表示
3. **スクリーンショットを撮る**: `browser_take_screenshot` でユーザーに見せる（ターミナル上で確認可能）
4. **ユーザーの回答はターミナルで受ける**: AskUserQuestion でフィードバックを収集
5. **次の質問**: 新しい HTML ファイルを書いて繰り返す。不要になったらブラウザはそのまま放置してよい

### デザインのコツ

- **ダークテーマ必須**（ユーザーのダークモード設定に合わせる）
- カスタム色は控えめに（背景: #1a1a2e 系、アクセント: #6c63ff 系）
- 選択肢は 2-4 個まで
- 質問の意図をページ上部に明記（「どのレイアウトが直感的？」等）
- ワイヤーフレームはシンプルに。ピクセルパーフェクトは不要
- 比較する時は split view（横並び）

## Phase 3: 構造化された質問

**必ず AskUserQuestion ツールを使用する**。会話形式の質問は使わない。

### ルール

- **1回の質問は 1問**（superpowers 方式。一問ずつ深掘りする）
- 各質問に **2-4 個の具体的な選択肢**
- 各選択肢に **pros/cons を簡潔に** 記載
- CLAUDE.md の既存パターンに沿った選択肢にする
- 複数選択（multiSelect）は本当に必要な場合のみ

### 質問の優先順位

1. **ブロッカー**: これが決まらないと他の判断ができない
2. **アーキテクチャ**: 後から変更すると影響が大きい
3. **スコープ**: 作業量に直結する
4. **UI/UX**: ユーザー体験に影響する
5. **詳細**: 実装時に判断しても問題ない

十分な情報が集まったら Phase 4 に進む。

## Phase 4: アプローチ提案

設計を固める前に、2-3 のアプローチを提案する：

- 各アプローチのトレードオフ（複雑性、拡張性、工数、リスク）を明示
- **推奨案を先に示し**、その理由を説明
- 会話的に提案し、ユーザーのフィードバックを受ける

```markdown
## アプローチ案

### A: [推奨] ○○○○
- トレードオフ: ...
- 推奨理由: ...

### B: ○○○○
- トレードオフ: ...

### C: ○○○○
- トレードオフ: ...
```

ユーザーがアプローチを選択（または修正を依頼）したら Phase 5 に進む。

## Phase 5: 設計提示 & 承認ゲート

選択されたアプローチに基づき、設計を提示する：

- 各セクションの長さはその**複雑さに比例**させる（シンプルなら数行、複雑なら 200-300 語）
- カバーすべき内容: アーキテクチャ、コンポーネント、データフロー、エラーハンドリング、テスト戦略
- **セクションごとに** 確認を取り、問題があれば戻って修正

### 設計の原則

- **分離と明確さ**: 各ユニットに1つの明確な目的、明確なインターフェース、独立したテスト可能性
- **YAGNI**: 不要な機能は設計から削除
- **既存パターンの尊重**: 既存コードベースのパターンに従う。無関係なリファクタは提案しない

**承認ゲート**: 全セクションの承認が得られてから Phase 6 に進む。

## Phase 6: スペック文書の出力 & セルフレビュー

### 文書の保存

承認された設計を以下のパスに保存：

`docs/specs/YYYY-MM-DD-<topic>-design.md`

（プロジェクト固有の保存先がある場合はそちらを優先）

### セルフレビュー

保存後、以下のチェックリストを実行：

1. **プレースホルダースキャン**: TBD、TODO、未完成セクション、曖昧な要件がないか
2. **内部整合性**: セクション間で矛盾がないか。アーキテクチャと機能記述が一致するか
3. **スコープチェック**: 1つの実装計画で扱える範囲か。分解が必要ではないか
4. **曖昧性チェック**: 2通りに解釈できる要件がないか。あれば1つに確定する

問題があればインラインで修正。

### 決定事項テーブル

スペック文書の末尾に、dig プロセスで決まった全決定事項を記録：

```markdown
## 決定事項

| # | カテゴリ | 項目 | 決定 | 理由 |
|---|---------|------|------|------|
| 1 | アーキテクチャ | ... | ... | ... |
```

## Phase 7: ユーザーによるスペックレビュー

> "スペックを `<path>` に保存しました。実装計画に進む前にレビューをお願いします。変更があればお知らせください。"

ユーザーの応答を待つ。変更要求があればスペックを修正し、セルフレビューを再実行。

## Phase 8: 実装ハンドオフ

スペック承認後、次のステップを提案：

> **スペック承認済み。実装に進みます。**
>
> 1. **`/write-plan`** — 詳細な実装計画を作成（bite-sized タスクに分解）
> 2. **`/task-decompose`** — 並列実行可能なサブタスクに分解（大規模変更向け）
> 3. **`/tdd`** — 直接 TDD で実装開始（小規模変更向け）
>
> どれで進めますか？

**ターミナルステートは実装スキルへのハンドオフ。** dig 自身はコードを書かない。

## 使用するツール

- **Read** --- コンテキストファイルの読み込み
- **Grep / Glob** --- 既存コードの調査
- **AskUserQuestion** --- 構造化された質問（**必須**）
- **Write** --- スペック文書・Visual Companion HTML の出力
- **Playwright MCP** --- Visual Companion（browser_navigate, browser_take_screenshot）

## 注意事項

- **ファイル編集はスペック文書と Visual Companion HTML のみ**: 実装コードには触れない
- **掘り下げは徹底的に**: 表面的な質問で終わらず、回答から派生する新たな曖昧点も追求する
- **既存パターンを尊重**: プロジェクトの既存アーキテクチャや規約に反する選択肢は、その旨を明示する
- **1問ずつ質問**: 複数の質問を一度に投げない。深く掘り下げる
